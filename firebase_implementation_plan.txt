PHASE 1: FIREBASE DATA STRUCTURE
================================================================================

FIRESTORE COLLECTIONS:

1. transactions (Platform Data)
   ├── Document ID: transaction_id (auto-generated)
   ├── Fields:
   │   ├── transaction_id: string (unique)
   │   ├── amount: number (in paise/smallest currency unit)
   │   ├── currency: string ("INR", "USD", etc.)
   │   ├── transaction_date: timestamp
   │   ├── status: string ("pending", "completed", "refunded")
   │   ├── customer_id: string
   │   ├── description: string
   │   └── created_at: timestamp
   │
   └── Example Document:
       {
         transaction_id: "TXN_001",
         amount: 10000,
         currency: "INR",
         transaction_date: 2024-01-15,
         status: "completed",
         customer_id: "CUST_123",
         description: "Product Purchase",
         created_at: 2024-01-15T10:30:00Z
       }

2. settlements (Bank Data)
   ├── Document ID: settlement_id (auto-generated)
   ├── Fields:
   │   ├── settlement_id: string (unique)
   │   ├── transaction_id: string (reference to platform transaction)
   │   ├── settled_amount: number (actual amount received)
   │   ├── settlement_date: timestamp
   │   ├── bank_reference: string
   │   ├── status: string ("processed", "pending", "failed")
   │   └── created_at: timestamp
   │
   └── Example Document:
       {
         settlement_id: "SET_001",
         transaction_id: "TXN_001",
         settled_amount: 10000,
         settlement_date: 2024-01-17,
         bank_reference: "BANK_REF_12345",
         status: "processed",
         created_at: 2024-01-17T02:15:00Z
       }

3. refunds (Refund Data)
   ├── Document ID: refund_id (auto-generated)
   ├── Fields:
   │   ├── refund_id: string (unique)
   │   ├── original_transaction_id: string (references transactions)
   │   ├── refund_amount: number
   │   ├── refund_date: timestamp
   │   ├── reason: string
   │   ├── status: string ("initiated", "processed", "failed")
   │   └── created_at: timestamp
   │
   └── Example Document:
       {
         refund_id: "REF_001",
         original_transaction_id: "TXN_010",
         refund_amount: 5000,
         refund_date: 2024-01-20,
         reason: "Customer Request",
         status: "processed",
         created_at: 2024-01-20T14:45:00Z
       }

4. reconciliation_reports (Output Reports)
   ├── Document ID: report_id (auto-generated)
   ├── Fields:
   │   ├── report_id: string
   │   ├── month_year: string ("2024-01")
   │   ├── matched_transactions: number
   │   ├── missing_in_bank: array of objects
   │   ├── missing_in_platform: array of objects
   │   ├── duplicates: array of objects
   │   ├── rounding_issues: array of objects
   │   ├── refund_errors: array of objects
   │   ├── total_variance: number
   │   ├── generated_at: timestamp
   │   └── status: string ("draft", "published")

5. assumptions (Configuration)
   ├── Document ID: assumptions_doc
   ├── Fields:
   │   ├── matching_date_tolerance_days: number (1)
   │   ├── amount_tolerance_paise: number (1)
   │   ├── currency_list: array
   │   ├── rounding_decimal_places: number (2)
   │   ├── settlement_delay_days: number (2)
   │   └── last_updated: timestamp

PHASE 3: MATCHING LOGIC ALGORITHM
================================================================================

STEP 1: DATA RETRIEVAL
function getReconData(month, year) {
  1. Fetch all transactions for month/year
  2. Fetch all settlements for month/year
  3. Fetch all refunds for month/year
  4. Load assumptions from config
}

STEP 2: BASIC MATCHING (transaction_id)
function basicMatch(transactions, settlements) {
  for each transaction:
    - Look for settlement with matching transaction_id
    - If found: Store as "matched"
    - If not found: Store as "missing_in_bank"
  
  for each settlement:
    - Look for transaction with matching transaction_id
    - If not found: Store as "missing_in_platform"
}

STEP 3: AMOUNT & DATE VALIDATION (Matched records)
function validateMatches(matches, assumptions) {
  for each match:
    - Get transaction amount and date
    - Get settlement amount and date
    
    IF transaction_amount != settlement_amount:
      IF difference <= assumptions.amount_tolerance_paise:
        → Flag as "rounding_issue"
      ELSE:
        → Flag as "amount_mismatch"
    
    IF ABS(transaction_date - settlement_date) > assumptions.matching_date_tolerance_days:
      → Flag as "date_mismatch"
}

STEP 4: DUPLICATE DETECTION
function detectDuplicates(transactions, settlements) {
  Create frequency map for transaction_id in transactions
  Create frequency map for transaction_id in settlements
  
  If count > 1 in transactions:
    → Flag as "duplicate_in_platform"
  
  If count > 1 in settlements:
    → Flag as "duplicate_in_bank"
}

STEP 5: REFUND VALIDATION
function validateRefunds(refunds, transactions) {
  for each refund:
    - Check if original_transaction_id exists in transactions
    - If not found:
      → Flag as "refund_error" / "orphaned_refund"
    - If found:
      → Verify transaction status can be refunded
}

STEP 6: CLASSIFICATION LOGIC
Classification Categories:
  ✅ MATCHED - transaction_id found, amounts within tolerance, dates within tolerance
  ❌ MISSING_IN_BANK - transaction exists but no settlement
  ❌ MISSING_IN_PLATFORM - settlement exists but no transaction
  ⚠️ AMOUNT_MISMATCH - matched but amounts differ beyond tolerance
  ⚠️ DATE_MISMATCH - matched but settlement date too far from transaction
  🔢 ROUNDING_ISSUE - amount mismatch within tolerance (0.01)
  🔁 DUPLICATE - transaction appears multiple times
  💸 REFUND_ERROR - refund references non-existent transaction
  ⚠️ TIMING_ISSUE - transaction in one month, settlement in another

PHASE 5: ASSUMPTIONS & TOLERANCES
================================================================================

ASSUMPTION 1: Matching Tolerance
├── Date Tolerance: ±1 day (can be ±2 days during month-end)
├── Rationale: Bank processes batches 1-2 days after transaction
└── Config: matching_date_tolerance_days = 1

ASSUMPTION 2: Amount Tolerance
├── Tolerance: ±1 paise (±0.01 INR)
├── Rationale: Rounding differences in currency conversion
└── Config: amount_tolerance_paise = 1

ASSUMPTION 3: Settlement Delay Window
├── Max Delay: 2 days
├── After 2 days: Flag as "late_settlement" or "missing_in_bank"
├── Rationale: Industry standard for bank settlement
└── Config: settlement_delay_days = 2

ASSUMPTION 4: Refund Matching
├── Refunds must reference valid transaction_id
├── Refund amount must be ≤ original transaction amount
├── Refund processed only if original transaction complete
└── Rationale: Standard financial controls

ASSUMPTION 5: Duplicate Definition
├── Same transaction_id appearing 2+ times in same dataset
├── Flag ONLY if amounts differ or settlement mismatch
├── Rationale: Prevent false positives from legitimate retries
└── Config: duplicate_check_strict = true

ASSUMPTION 6: Time Period Boundaries
├── Month cutoff: First day of month to last day (23:59:59)
├── But: Settlements can arrive up to 2 days into next month
├── Rationale: Allow for processing delays
└── Config: include_next_month_settlements = true (within 2 days)

ASSUMPTION 7: Currency
├── Single currency testing: INR (Indian Rupees)
├── Stores amounts in PAISE (smallest unit × 100)
├── Rationale: Avoid floating-point precision issues
└── Config: base_currency = "INR", unit_multiplier = 100

ASSUMPTION 8: Data Integrity
├── transaction_id is globally unique
├── settlement_id is globally unique
├── refund_id is globally unique
├── No circular dependencies
└── Rationale: Enable reliable matching
